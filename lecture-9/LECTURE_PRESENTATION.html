Recap: coupling

Coupling = how much one part depends on another’s details (schemas, call order, deployment timing).

  • Strong coupling → small provider changes ripple to many consumers
  • Architecture views should show dependency direction and hotspots (shared DB, DTOs)

Recap: compatibility

Syntactic

Wire shape parses: fields, types, required vs optional.

Semantic

Meaning: validation, idempotency, error code semantics—can break without JSON renames.

Backward compatibility is the usual goal for public APIs and long-lived partners.

The age of continuity

Release cadence shifted from years (e.g. major desktop OS) to continuous updates in the cloud.

  • Users may no longer perceive a “version” — each visit can hit a slightly newer system
  • Industry example (lecture): providers moved from ~10 releases / 10s to ~1 / second scale (circa 2011→2014 narrative)
  • Challenge: high frequency without sacrificing correctness and other qualities

CityBite’s dinner spikes make safe, frequent deploys a business requirement—not only an engineering preference.

Deployability metrics — latency & throughput

Deployability can be measured like other quality attributes.

  • Latency (forward): dev commits → user sees feature or bugfix in production
  • Latency (backward): user feedback → triage → fix → user accepts the improvement
  • Throughput: releases per hour / day / month — from “once every few years” to many per day

Deployability metrics — time, cost, pipeline

  • Downtime: how long are users blocked during upgrade? (maintenance page vs seamless)
  • Deadlines: marketing launch (e.g. holiday) — slip = revenue risk
  • Human cost: one-click vs multi-team “release weekend” coordination
  • Infrastructure cost: laptop build vs cluster of VMs for parallel compile + hours of tests

Huge mono-repos: “wait until tomorrow’s nightly” limits developer feedback—deployability of the architecture and build.

Every release: opportunity and risk

Opportunity

Features, fixes, performance, revenue, retention, competitive catch-up.

Risk

New bugs/failure modes, capacity cost, support & training load, bad deploy, unrecoverable state (data loss).

Michael T. Nygard — Release It!; operators may fear change for good reason—design for rollback.

No change ≠ no risk

  • Only unused software never needs to change
  • Goal: minimize risk of change — not “minimize change” forever
  • Small deltas, frequent practice, reversible steps (like DB transactions: commit / rollback)

“Weekends have two days: Saturday deploy, Sunday revert” — plan time buffer and recovery strategy.

Big Bang vs continuous flow

  • Classic waterfall handoff: dev “done” → ops installs; first real test is often production load
  • “If it hurts, do it more often” — reduce batch size and rehearse the transition
  • Continuous: build & operate in parallel; ops feedback loops to dev; red deploys are expected and fixed fast

Speed vs quality — automation reframes the tradeoff

  • Old assumption: fast shipping ⇒ low quality; high quality ⇒ slow
  • Continuous delivery cannot sacrifice quality every push—would live in a broken state
  • Automated QA + repeatable packaging ⇒ high throughput and high confidence together

The pipeline that builds software is software — version, review, and test it too.

Software production pipeline

Input: source repo → output: immutable artifact (image) ready for users.

  • Stages: build (compile + unit tests + static analysis) → integrate → deeper tests (perf, acceptance)
  • Release decision — human or automated gate → deploy → smoke / readiness in prod
  • Word that repeats in the chapter: test — encode expectations so machines can enforce them every run

CI · Continuous Delivery · Continuous Deployment

TermExtends to…
Continuous IntegrationFrequent merge; automated build + unit/integration to a runnable integrated product
Continuous DeliveryArtifact fully tested and packaged; manual approval before production deploy
Continuous DeploymentAutomatic deploy when checks pass — needs exceptional test coverage; mobile stores add delay

High quality at high speed — practices

  • Everyone sees the pipeline; don’t push on broken build; watch until green; don’t go home on red
  • Small commits; fast feedback on push; nightly heavier suites when needed
  • Reproducible builds — snapshot dependencies; immutable released images (no hot-edits in prod)
  • No “comment out failing tests”; time-box fix then revert; forbid un-tracked prod edits—fix at source

Jez Humble & David Farley — Continuous Delivery.

Testing in production always exists; pre-release tests decide how often users are first to find bugs.

Release strategies

  • Big Bang / plunge — everyone switches; risky rollback
  • Blue / green — instant traffic flip; keep old stack for rollback (2× capacity cost)
  • Shadow / dark launch — new version consumes real input; output hidden; compare behavior
  • Pilot / canary / gradual / A/B — ramp trusted or random users; measure metrics before full cutover

Gradual strategies require multiple active versions — limits how many trains you can run in parallel.

Virtualization

  • Hypervisor multiplexes guest OS + apps over physical hardware — elasticity, migration, isolation
  • Hosted hypervisor (e.g. laptop: host OS + VirtualBox) vs native hypervisor on bare metal (datacenter)
  • Live migration — move VMs between hosts for maintenance or failure—impossible with pets-on-metal alone

Containers in VMs — common cloud pattern: rent VM, run many containers for density and parity.

Images, snapshots, migration

  • Image — frozen filesystem + metadata to boot a known state (VM or container)
  • Snapshots / checkpoints — capture RAM + disk for resume or disaster recovery
  • Repository — versioned artifacts; digest = what actually ran in the incident postmortem

Virtual hardware as software

  • CPU/RAM/disk/network allocation lives in versioned config (Terraform, Helm, K8s YAML)
  • Never mix production and test environment definitions—same discipline as application code
  • Inverted hypervisor (advanced): one logical machine spread over many hosts for huge workloads

Deployability

Deployability = how easily and safely you can move a change from commit to production—and roll back if needed.

  • Automation — CI builds tests and artifacts; CD promotes known-good images
  • Frequency & blast radius — smaller diffs, feature flags, canary or rolling updates
  • CityBite pain — SSH to long-lived VMs, dinner-rush deploys, minutes of partial downtime during restarts

Architecture choices (state, sessions, disk writes) directly constrain how often you can deploy.

Portability

Portability = the same application artifact runs in dev, staging, and prod without one-off hacks per host.

  • Configuration — environment variables, mounted config maps—not secrets baked into images
  • Filesystem — avoid hardcoded OS paths (C:\...) and “always writable cwd” assumptions
  • Dependencies — pinned libraries; reproducible builds from lockfiles / image layers

Containers help portability but do not remove bad path habits—see Example 2 (DATA_DIR).

Containers (high level)

  • Image — immutable template: OS userland slice, runtime, app files, metadata
  • Container — running instance from an image: isolated process tree, namespaces, cgroup limits
  • Orchestrator (e.g. Kubernetes) — schedules replicas, replaces failed nodes, exposes Services/Ingress

Treat the image digest as “what actually ran” when debugging production incidents.

Images, layers, immutability

  • Layers — cache dependency installs; smaller layers = faster pull on new nodes
  • Immutable servers — replace containers instead of patching live filesystems ad hoc
  • Secrets — inject at runtime (K8s Secret, cloud secret manager); never docker commit credentials into an image

Multi-stage builds keep compile toolchains out of the final runtime image (smaller attack surface).

Twelve-factor style (selected)

Config & port

DATABASE_URL, LOG_LEVEL, PORT from the environment—what Fargate/EKS inject.

Logs & processes

Stream logs to stdout/stderr; one main process; handle SIGTERM for graceful drain.

Liveness vs readiness

Liveness

“Is the process wedged?” — failure → restart the container.

Readiness

“Can this pod take traffic?” — DB down → remove from Service until recovered.

Wrong probes cause restart storms or send traffic to pods that are still starting—design thresholds carefully.

State: CityBite menu photos

  • App pods should be disposable—scale out/in without copying disk by hand
  • Menu JPEGs — today on VM disk; in K8s use PVC + DATA_DIR or S3 (compare in assignment)
  • Postgres — typically managed outside the cluster; pods get DATABASE_URL

Example 2 shows why ./uploads fails on read-only or ephemeral container roots.

Pets vs cattle & pitfalls

  • Pets — snowflake servers with manual fixes; hard to reproduce
  • Cattle — replaceable nodes and pods; config and data live outside the disposable unit
  • Pitfalls — sticky sessions in memory, local file sessions, writing outside mounted volumes

Moving CityBite off VMs is as much culture (IaC, runbooks) as it is Docker syntax.

Security & ops hygiene

  • Run as non-root in the container where possible; read-only root FS + explicit writable paths
  • Scan images for CVEs; patch base images on a schedule
  • NetworkPolicy / service mesh (optional) to limit lateral movement between namespaces
