Recap: interfaces & contracts

Good interfaces are minimal (only what clients need), stable (avoid breaking changes), and clear (consistent naming and semantics).

  • Abstraction / information hiding — expose behavior, hide implementation details
  • Contracts — preconditions (what must hold before a call), postconditions (what holds after)
  • Multiple implementations behind one interface (e.g. cache backends) = real reusability

Recap: API design

  • Consistency — patterns for naming, errors, and resources across endpoints
  • Discoverability — docs, predictable URLs or RPC names
  • Backward compatibility — don’t surprise existing clients when you extend

Lecture 6 assignment tied this to exposing a library as an HTTP/REST API (layers, request/response shapes).

LibraryFramework
Control flowYou call itIt calls you (inversion)

Same component can be packaged as a library or hosted inside a framework — the interface story changes.

Recap: evolution & versioning

  • Extend with optional parameters (defaults) instead of breaking signatures
  • Deprecate clearly (warnings, docs), remove in a later major release
  • Semantic versioning: MAJOR = breaking, MINOR = compatible features, PATCH = fixes

Plan a migration path when interfaces must change.

What is composition?

Composition = building a system by assembling smaller components rather than one monolithic block.

  • Each component has a clear responsibility
  • Components are connected by connectors
  • Architecture ≈ components + connectors

You reuse and replace parts; the whole is defined by how parts interact.

Connectors

A connector is the mechanism that allows communication between components.

  • Defines protocol, direction, and often sync vs async
  • Same component can use different connectors to different neighbors
  • Choosing the wrong connector affects coupling, latency, and failure modes

Connector types (overview)

TypeTypical useOften
Direct callSame process, in-memorySync
RPC / RESTServices over networkSync (request/response)
Message queueDecouple producer/consumerAsync
Event busMany subscribers, pub/subAsync
Shared DBImplicit coupling via dataSync read/write

Message passing

Synchronous

Caller sends a message and waits for a reply.

Simple mental model; tight temporal coupling; blocking under load.

Asynchronous

Caller sends and continues; consumer processes later.

Better decoupling and scale; harder reasoning and debugging.

When to use sync vs async?

  • Sync when you need an immediate result to proceed (e.g. validate payment before shipping)
  • Async when work can be deferred (e.g. send email, generate report)
  • Async often uses queues or events to absorb spikes and isolate failures

Composition patterns

  • Pipeline — output of A is input of B (sequential stages)
  • Layered — upper layer calls lower layer only
  • Blackboard / shared state — components read/write shared data (use with care)
  • Microservices / SOA — services composed via APIs and messaging

Service composition

Multiple services cooperate to fulfill one business capability.

  • Each service exposes a contract (API or events)
  • Composition can be orchestrated or choreographed
  • Cross-cutting concerns: auth, tracing, idempotency, timeouts

Orchestration

One component (the orchestrator) controls the flow: it calls other services in order and handles branching.

  • Pros: Flow is visible in one place; easier to change sequence
  • Cons: Orchestrator can become a bottleneck; couples flow to one component

Choreography

No central coordinator: each service reacts to events (or messages) and may publish new ones.

  • Pros: Loose coupling; easy to add new subscribers
  • Cons: Global flow is implicit; harder to trace and test end-to-end

Orchestration vs choreography

OrchestrationChoreography
ControlCentralDistributed
VisibilityHigh (one place)Lower (many handlers)
ScalingOrchestrator mattersWorkers scale independently
Good forClear workflowsMany reactive consumers

Event-driven architecture

  • Event — something happened (past tense: OrderPlaced)
  • Event bus / broker — delivers events to subscribers
  • Enables choreography, async processing, and integration without tight API coupling
  • Watch for: ordering, duplicates, eventual consistency, schema evolution

Architectural connectors

At the architecture level, connectors are not just “function calls”:

  • They carry quality attributes: reliability, latency, security
  • They appear in deployment and runtime views
  • Document them in diagrams (REST, AMQP, gRPC, WebSocket, …)

Takeaways

  • Systems are composed of components linked by connectors
  • Choose connector type and sync/async to match requirements and trade-offs
  • Orchestration centralizes flow; choreography distributes it via events
  • Event-driven design supports scalable, loosely coupled composition
